[v2.13.145] DRIFT PERFECTO POR ETIQUETA + sub-job default 10 → 25

Usuario observó que el batching de 10 generaba confusión y preguntó si era
posible imprimir "todo de un tirón" o "al unitario en todo caso". Auditoria
senior detecto bug latente:

PROBLEMA (≤v2.13.144):
  Dentro de un sub-job de N etiquetas, _calcularOffsetEfectivoParaPrint() se
  llamaba UNA VEZ al inicio → TODAS las N etiquetas compartían el MISMO
  offsetY. La etiqueta #1 quedaba bien centrada pero la #10 desfasada por
  10 × driftDots = ~2 mm (drift no compensado dentro del batch).

FIX:
  Refactor de _buildTSPLEtq para construir TSPL multi-etiqueta:
  - 1 header global (SIZE/GAP/DENSITY/SPEED) al inicio del job
  - Loop N veces: CLS + contenido(offsetY_i) + PRINT 1,1
  - offsetY_i = clamp(offsetBase + driftDots × (printsBase + i))

  Cada etiqueta del sub-job tiene su offsetY EXACTO según su posición real
  en el rollo. Drift compensa por print individual, no por sub-job.

CONSECUENCIA:
  Sub-jobs grandes ya no acumulan error → default subJobSize subido 10 → 25.
  Para imprimir 25 etiquetas alcanza 1 sub-job (1 HTTP). Para lotes
  bigger los trigger sigue procesando en sub-jobs de 25 c/u.

PAYLOAD:
  100 etiquetas en 1 sub-job = ~172 KB base64 → muy debajo del limite
  PrintNode (100 MB). Logo bitmap 828 bytes binarios x N etiquetas + texto.

MEMBRETES:
  Membretes ME/WH ya estaban OK — cada membrete = 1 job HTTP individual →
  _calcularOffsetEfectivoParaPrint() se llamaba por print = drift perfecto.
  No requieren refactor.
